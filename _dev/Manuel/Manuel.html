Présentation

Cette application est pensée pour simplifier la gestion de l’installation du travail lorsqu’on développe plusieurs projets de front.

Elle se présent comme un panneau affichant chaque projet avec, pour chaque projet, un bouton permettant d’installer et de préparer le travail.

Cette instllation peut consister à :

  • ouvrir le dossier du projet dans le finder, dans une fenêtre personnalisée (taille, position, type de vue),

  • actualiser un fichier versionné et mettre l’ancien dans un dossier d’archives,

  • lancer un script qui va chercher une information sur le net,

  • ouvrir un fichier dans LibreOffice,

  • ouvrir un dossier de code dans VSCode,

  • et bien d’autres choses.

On appelle toutes ces opérations des services qu’on peut appliquer aux projets.

Prise en main

Après avoir lancé l’application, on se retrouve devant un tableau de bord qui permet d’ajouter ses projets en définissant pour chacun d’eux les Les services dont on a besoin.

Chaque projet se présente sous forme d’une carte (un peu comme dans Trello) qui possède en titre le nom du projet et contient la liste des différents services qui lui ont été affectés.

~ “Affecter des services” est l’expression consacrée pour dire qu’on utilise des Les services pour un projet. ~

Chaque carte est composée de trois parties :

  • l’entête, avec le titre du projet et quelques informations le concernant,

  • une partie supérieure contient tous les services qui seront activés lorsque l’on cliquera sur le bouton “Démarrer le travail”,

  • une partie inférieure contient les boutons qu’on peut utiliser à l’occasion pour des traitements particuliers.

Panneau de nouveau projet

Quand on clique sur le "+" qui se trouve à gauche en bas de l’interface, un panneau s’ouvre pour définir le nouveau projet.

La partie supérieure permet d’entrer les informations sur le projet :

  • lui donner un titre,

  • indiquer son dossier de référence (un projet doit toujours avoir un dossier),

La partie inférieure permet de choisir les Les services par simple glisser-déposer.

Exemple : pour le service “Ouvrir dans le Finder”, une fois qu’on l’a glissé sur le projet, une fenêtre s’ouvre, demandant d’ouvrir la fenêtre dans le Finder, de la positionner et de la mettre ans la vue désirée (colonnes, lignes, etc.). Une fois que c’est fait, on clique sur “OK” et l’application va automatiquement prendre les informations sur la fenêtre au premier plan dans le finder.

Les services

Définition des services

Il existe deux grandes catégories de services :

  • les services personnalisés

  • les services communs

Services personnalisés

On peut basculer des services communs aux custom-services grâce au bouton “Services communs”.

Services communs

Les services communs ne s’ajoutent pas aux projets, ils sont toujours accessibles par eux. Le premier service commun est par exemple “Ouvrir le dossier du projet”

On affiche les services communs simplement en cliquant sur la fiche d’un projet. En même temps qu’il se met en sélection, la liste des services communs s’affiche à droite de l’interface.

On peut basculer des services personnalisés aux services communs grâce au bouton “Services personnalisés”.

Cette partie décrit tous les services qu’on peut utiliser pour un projet.

Collecte informations

Note : Ce fichier définit des processus qui seront en pratique plus tard. Pour le moment, l’utilisateur remplit des fiches projet qui le définissent explicitement.

Chaque service a besoin de collecter des informations. Pour ce faire, il y a :

  • les informations directement demandées à l’utilisateur (p.e. : titre à donner à un service)

  • les informations récupérées automatiquement par script (p.e. : information sur la fenêtre dans le Finder)

- un service est déplacé sur la carte du projet
- le service demande ses informations, c'est une suite d'opérations
  simples comme :
    - demander le titre à donner
    - lancer le script de collecte d'information fenêtre
    - demander confirmation
    - créer le service

Le service définit les informations dont il a besoin. Par exemple, pour l’ouverture d’une fenêtre Finder :

  • le path au dossier

  • la position et la taille

  • le type de vue

L’intégralité des services est définie dans le fichier frontend/js/ServicesData.js

Les Les services connaissent toujours deux états :

  • l’état en tant que service absolu, qui peut être incorporé à un projet

Par exemple, on peut avoir :

  • le service abstrait open-finder-window qui peut ouvrir une fenêtre de Finder,

Paramètres fixes et paramètres dynamiques

Tous les services fonctionnent avec des paramètres fixes défines au moment de l’affectation d’un service à un projet. La première de ces paramètres fixes est le nom propre au projet du service. Par exemple, si le nom (name) du service est “Versionner un fichier”, le nom défini pour le projet par l’utilisateur au moment de l’affectation du service peut devenir “Versionner le manuscrit”.

Certains services nécessitent aussi des paramètres dynamique qui, elles, sont définies à la volée au moment de l’utilisation du service. Par exemple, au moment de versionner le manuscrit, Board va demander s’il faut incrémenter le numéro de path, de version mineur ou de version majeure.

Définition des paramètres fixes

Ces paramètres sont définis dans le fichier frontend/js/ServiceData.js. Si ces données sont modifiées, il faut lancer le script update.command à la racine de l’application.

Une définition ressemble à :

, params: [
    {id: 'window', q: null, value: null, type: 'finder-window', required: true},
    {id: 'sidebar', q: "Voulez-vous la sidebar ?", value: null, type: 'boolean', required: false}
  ]

id

fournit l’identifiant de l’élément DOM qui va contenir la valeur (#<id>.value)

q

définit la question qui sera posée à l’utilisateur pour définir le paramètre.

type

définit le type du champ d’édition(select, text-field, etc.). Voir tous les types de champ d’édition

Autres valeurs qui peuvent être définies en fonction du type`

values

définit les valeurs pour un menu select. Peut être [x, y, …] ou [[x, "nom x"], [y, "nom y"], …]

Ces paramètres doivent être définis dans l’ordre où ils seront consommés par le script backdend qui va les jouer.

Pour mémoire seulement, on peut les définir dans paramsOrder :

, paramsOrder: ['path', 'x', 'y', 'width', 'height', 'sidebar-width', 'type-view', 'show-sidebar']

Script Backend

On appelle Script Backend le script placé dans le dossier backend/scripts/ qui va procéder à l’opération propre au service. Le script, par exemple, qui va changer la version du fichier.

Le nom du script ne peut pas être choisi au hasard, il correspond à l'`id` du service (par exemple "versionner-fichier") camelisé ("VersionnerFichier"). Son extension correspond bien entendu à son langage.

Dans ce script, qui peut être un script AppleScript, un script ruby, bash ou autre langage, les paramètres sont reçus à l’aveugle et en string.

À l’aveugle

Ils sont connus par leur ordre, sans aucune indication de ce qu’ils sont

En string

Toutes les valeurs sont des chaines de caractères, mais "true" ou "12".

Ordre des paramètres dans le script

L’ordre de réception des paramètres est celui défini par l’ordre des définitions des paramètres dans le fichier ServiceData.js. Le problème est que certains type de champ retourne plusieurs valeurs. C’est le cas typique, par exemple, pour le type finder-window qui retourne la position (2 valeurs), la taille (2 valeurs), le type de vue, etc. pour le seul paramètre window. Pour parlier ce problème, on peut renseigner la propriété d’aide paramsOrder avec l’ordre des paramètres.
